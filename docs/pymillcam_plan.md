# PyMillCAM — Architecture & Planning Document

**A Python-Based 2D/2.5D CAM Tool**

Version 0.4 — April 2026 | Status: Initial Planning / Brainstorming

---

## 1. Vision & Goals

PyMillCAM aims to fill the gap between simple but limited tools like Estlcam and powerful but complex tools like Fusion 360's CAM or FreeCAD's Path workbench. The target is a Python-based, open-source 2D/2.5D CAM tool that is wizard-driven for beginners yet fully editable for experienced users.

### 1.1 Design Principles

- **Wizard-first, tree-editable:** Wizards guide common operations, but every parameter lands in an editable operations tree. Power users can skip wizards entirely.
- **Visual clarity:** Every setting should have a visual representation. Z stack diagrams, toolpath previews, origin markers, and fixture overlays make the invisible visible.
- **Safety by default:** Sanity checks for Z reachability, travel limits, fixture collisions, and toolpath validation run automatically before G-code generation.
- **Cascading configuration:** Machine defaults → Project settings → Operation overrides. Set it once at the right level, override only when needed.
- **Separation of concerns:** Geometry engine, toolpath engine, and post-processor are independent layers. The UI is decoupled from the computation.
- **Interoperability:** Import FreeCAD tool libraries and LinuxCNC tool tables. Export G-code for UCCNC, Mach3, GRBL, LinuxCNC, and others via configurable post-processors.

### 1.2 Target Use Cases

- Sheet cutting (aluminum, plastic, wood) with profile operations and tabs
- Pocketing for enclosures, inlays, and recesses
- Drilling patterns, including bounded patterns (e.g., ventilation grids)
- Surface planing / facing operations
- Engraving and V-carving of text and geometry
- 2.5D contour machining (stepped depths)
- Tiling / panelization for parts larger than the machine envelope

### 1.3 Inspirations

- **Estlcam:** Click-on-geometry-to-assign-operations simplicity. Minimal UI. Quick path from DXF to G-code.
- **CamBam:** Machining Operations (MOP) concept, hole pattern generators with bounding regions, scripting extensibility, balance of simplicity and power.
- **FreeCAD CAM:** Tool library format (JSON-based .fctb/.fctl), post-processor architecture, open-source ecosystem.

## 2. System Architecture

The system is organized into three computational layers plus a UI layer and persistent data stores. Each layer has a well-defined responsibility and communicates through clean interfaces.

### 2.1 Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│  UI Layer (PySide6)                                         │
│  ┌─────────────┐ ┌────────────┐ ┌──────────┐ ┌────────────┐│
│  │  Viewport   │ │ Operations │ │Properties│ │  Wizards   ││
│  │  (2D/2.5D)  │ │ Tree       │ │ Panel    │ │  (Dialogs) ││
│  └─────────────┘ └────────────┘ └──────────┘ └────────────┘│
└─────────────────────────────────────────────────────────────┘
        │                    │                  │
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: Project Model (Data Layer)                        │
│  Stock, Geometry, Tools, Operations, Fixtures, Settings     │
└─────────────────────────────────────────────────────────────┘
        │
┌─────────────────────────────────────────────────────────────┐
│  Layer 2: Toolpath Engine                                   │
│  Profile, Pocket, Drill, Engrave, Surface, Pattern Gen      │
└─────────────────────────────────────────────────────────────┘
        │                    │
┌─────────────────────┐ ┌───────────────────────────────────┐
│  Layer 3: Post-Proc │ │  Persistent Data                  │
│  UCCNC, Mach3, GRBL │ │  Machine Library, Tool Library,   │
│  LinuxCNC, Custom   │ │  Material DB, App Preferences     │
└─────────────────────┘ └───────────────────────────────────┘
```

### 2.2 Layer 1 — Project Model (Data Layer)

The project model is a pure data layer with no UI dependencies. It uses Pydantic models for validation and serialization. The project file is saved as JSON.

#### 2.2.1 Core Data Models

**Project:** Top-level container holding all data for a single CAM job. References a machine definition and contains stock, geometry, operations, fixture setup, and project-level settings.

**Stock:** Material dimensions (X, Y, Z thickness), material type (references material database), and coordinate system settings (Z zero reference: top/bottom of stock, XY origin position).

**GeometryLayer:** A collection of 2D geometry entities (imported from DXF, or generated by pattern tools). Each layer has a name, color, and visibility flag. Entities are Shapely geometry objects (LineString, Polygon, Point, MultiPoint, etc.).

**Operation:** An abstract base with concrete subtypes (ProfileOp, PocketOp, DrillOp, EngraveOp, SurfaceOp, ContourOp). Each operation references geometry paths, a tool controller, and operation-specific parameters.

**ToolController:** Binds a tool from the library to an operation with runtime parameters: spindle RPM, feed rates, tool stickout. Multiple operations can share a tool controller if they use the same tool with the same settings.

**FixtureSetup:** Spoilboard thickness, fixture/jig type and height, clamp positions and dimensions. Used for Z budget calculation and collision checking.

#### 2.2.2 Configuration Cascade

Settings follow a three-level inheritance: Machine Defaults → Project Settings → Operation Overrides. Each level can override any parameter from its parent. In the UI, inherited values are shown in a lighter color with a "reset to default" option.

| Setting | Machine Default | Project Override | Operation Override |
|---|---|---|---|
| Units | mm | Inherit | N/A |
| Z zero reference | top_of_stock | bottom_of_stock | Inherit |
| Safe height | 15.0 mm | Inherit | 25.0 mm (clamp clearance) |
| XY origin | front_left | center | Inherit |
| Post-processor | uccnc | Inherit | N/A |

### 2.3 Layer 2 — Toolpath Engine

The toolpath engine is a pure-Python library that takes project model data and computes toolpaths. It has no UI dependencies and can be used headlessly for batch processing. The output is an intermediate representation (IR) — a structured sequence of machine instructions, not G-code.

#### 2.3.1 Intermediate Representation (IR)

The IR is a list of instruction objects representing abstract machine moves. Each instruction carries the information needed for any post-processor to generate correct G-code.

```python
class IRInstruction:
    type: str          # rapid, feed, arc_cw, arc_cc, dwell,
                       # spindle_on, spindle_off, tool_change,
                       # coolant_on, coolant_off, comment, macro
    x, y, z: float     # target position (optional per type)
    f: float            # feed rate (for feed/arc moves)
    s: int              # spindle RPM (for spindle_on)
    i, j: float         # arc center offsets (for arc moves)
    tool_number: int    # for tool_change
    comment: str        # for comment instructions
    macro_name: str     # for macro instructions (e.g. tool_probe)
```

#### 2.3.2 Toolpath Algorithms

- **Profile:** Offset curves (inside/outside/on-line), lead-in/lead-out (arc, tangent, direct), ramp entry options (helical, linear, plunge), tab generation with multiple styles (rectangular, triangular/tapered, thin-web), auto-placement or manual positioning, multi-depth stepping.
- **Pocket:** Zigzag, spiral, and offset-based strategies. Configurable stepover percentage, direction (climb/conventional with clear visual indicator), island detection, ramp entry (helical or linear — avoids plunging into material), multi-depth with optional finishing pass at reduced stepdown.
- **Drill:** Simple drill, peck drill, chip-break cycle. Supports ordered point lists, optimized drill path (nearest-neighbor or TSP heuristic).
- **Surface/Facing:** Full-stock-area pocket variant with optimized parameters for facing. Configurable overlap, direction, boundary offset for edge cleanup.
- **Engrave:** Follow-the-path engraving at a fixed depth. V-carve with depth computed from geometry width and V-bit angle.
- **Contour 2.5D:** Waterline-style stepped contour at multiple Z levels. Useful for 3D-ish shapes from 2D profiles.
- **Pattern Generator:** Creates geometry (not toolpaths) for common patterns: rectangular grid, hexagonal grid, circular array, text. Patterns are clipped to a bounding region. Output feeds into drill or profile operations.

#### 2.3.3 Key Libraries

| Library | Purpose | Notes |
|---|---|---|
| Shapely | 2D geometry operations | Offsets, booleans, clipping, buffering |
| ezdxf | DXF import/export | Read geometry layers and entities |
| clipper2 (pyclipper) | High-performance polygon offsetting | Better than Shapely for toolpath offsets |
| numpy | Numeric computation | Arc interpolation, point arrays |
| fonttools / HarfBuzz | Text to geometry conversion | For engraving text features |

### 2.4 Layer 3 — Post-Processor

The post-processor transforms the IR into G-code for a specific CNC controller. Each post-processor is a Python class implementing a simple interface. The machine definition specifies which post-processor to use and provides custom macros for machine-specific operations.

#### 2.4.1 Post-Processor Interface

```python
class PostProcessor(Protocol):
    def header(self, project: Project) -> list[str]
    def footer(self, project: Project) -> list[str]
    def rapid(self, x, y, z) -> str
    def feed(self, x, y, z, f) -> str
    def arc_cw(self, x, y, i, j, f) -> str
    def arc_ccw(self, x, y, i, j, f) -> str
    def spindle_on(self, rpm) -> str
    def spindle_off(self) -> str
    def tool_change(self, tool_num, macro) -> list[str]
    def dwell(self, seconds) -> str
    def comment(self, text) -> str
    def macro(self, name, params, template) -> list[str]
```

#### 2.4.2 Supported Controllers

| Controller | Priority | Key Differences |
|---|---|---|
| UCCNC | Primary | Standard G-code, M6 tool change, supports G53 |
| Mach3 | Primary | Similar to UCCNC, different safe-position handling |
| GRBL | Secondary | No tool length comp, limited canned cycles |
| LinuxCNC | Secondary | Full canned cycle support, O-word subroutines |

#### 3.2.1 Parametric Tool Visualization

Each tool is rendered as a 2D cross-section profile graphic, generated programmatically from the tool's geometry parameters. This eliminates the need for separate image files and ensures the visual always matches the actual dimensions. The rendering supports all tool shape types:

- **Endmill:** Rectangular cutting profile with shank. Diameter, flute length, and shank diameter are shown with dimension annotations.
- **Ball nose:** Hemispherical tip transitioning to cylindrical shank.
- **V-bit:** Conical tip with angle annotation, transitioning to shank.
- **Drill:** Pointed tip with angle, fluted body, shank.
- **Chamfer / bullnose:** Angled or radiused tip variants.

The tool graphic appears in the tool library browser (as an icon for quick visual identification), in the tool selection step of wizards, in the properties panel when a tool controller is selected, and as a small overlay in the viewport corner during simulation playback. The graphic is drawn using Qt's QPainter, rendered to scale with key dimensions labeled.

### 3.3 Material Database

A built-in collection of materials with default cutting parameters. Used by wizards to suggest starting feeds and speeds when a tool + material combination is selected.

| Category | Examples | Key Properties |
|---|---|---|
| Aluminum | 6061-T6, 7075, cast | Hardness, chip-load recommendations |
| Plastics | Acrylic, HDPE, Delrin, POM | Melting point awareness, speed limits |
| Wood | Hardwood, softwood, plywood, MDF | Grain direction, dust extraction notes |
| Composites | FR4, carbon fiber, G10 | Tool wear warnings, dust hazard notes |
| Foam / Wax | Machinable wax, rigid foam | High-speed, aggressive parameters |

### 3.4 Application Preferences

Global user preferences that apply across all projects: default units (mm/inch), UI theme, default machine selection, viewport colors, auto-save interval, and the default post-processor.

## 4. Z Stack Model & Safety Checks

One of the most important safety features. The real usable Z travel is not the machine's mechanical limit — it's what remains after accounting for tool stickout, fixtures, spoilboard, and stock.

### 4.1 Z Budget Concept

```
Machine Z travel:              80.0 mm  (mechanical limit)
─────────────────────────────────────────────────────────
Spindle nose at Z home (top of travel)
  − Tool stickout:              40.0 mm  (tool + collet)
  − Safe clearance:              2.0 mm  (above stock)
  − Stock thickness:             6.0 mm
  − Fixture/jig height:         15.0 mm  (vacuum table etc.)
  − Spoilboard thickness:       12.0 mm
─────────────────────────────────────────────────────────
Total needed from nose:        75.0 mm
Remaining margin:               5.0 mm  ← real safety buffer
```

### 4.2 Pre-Flight Safety Checks

Before G-code generation, the system runs three automated checks:

- **Reachability Check:** Can the tool physically reach the deepest cut? Compares tool stickout against the total Z distance from spindle nose to the lowest cutting depth through the full fixture stack.
- **Safe Height Check:** Does the safe travel height, when combined with the tool length, stay within the machine's upper Z travel limit? This catches the common crash where a rapid-up move slams into the Z limit switch.
- **Fixture Collision Check:** Do any rapid moves at safe height pass over clamp positions? Compares the 2D bounding boxes of rapid-traverse segments against clamp/fixture positions and heights.

Additionally, XY travel validation checks that all toolpath coordinates (including the offset from the work origin to machine origin) fall within the machine's X and Y travel range.

### 4.3 Fixture Setup Data

```json
"fixture_setup": {
  "spoilboard_thickness": 12.0,
  "fixture_name": "Aluminum vacuum table",
  "fixture_height": 15.0,
  "fixture_clearance_height": 20.0,
  "clamps": [
    {
      "x": 10, "y": 50,
      "width": 30, "length": 20,
      "z_height": 18,
      "label": "front-left clamp"
    }
  ]
}
```

## 5. User Interface Design

The UI is built with PySide6 (Qt6 for Python) and uses a dockable-panel layout. The design prioritizes visual feedback — every parameter change should be immediately reflected in the viewport.

### 5.2 Operations Tree

The operations tree is the primary navigation and editing interface. It uses a standard Qt QTreeView with a custom model. Each operation node shows its estimated machining time, and tool change pauses are shown as explicit separator nodes with their own time cost. This makes the impact of operation ordering immediately visible.

```
▼ Operations                            [Total: 47:32]
    Surface plane (6mm flat)             [  8:15]  🔧 T1
    ── tool change ──                    [  1:30]
    Profile cut (3mm flat)               [ 12:44]  🔧 T2
    Pocket - display (3mm flat)          [ 18:20]  🔧 T2
    ── tool change ──                    [  1:30]
    Drill - mounting (2mm drill)         [  5:13]  🔧 T3
```

Key behaviors:

- Drag-and-drop to reorder operations (execution order = tree order)
- Tool change nodes auto-insert/remove as operations are reordered — grouping same-tool operations eliminates a tool change and visibly reduces total time
- Multi-select geometry paths in viewport, right-click to assign to an operation
- Double-click an operation to open its wizard in edit mode
- Drag geometry entities between operations to reassign
- **Duplicate Operation:** Creates an exact copy preserving all geometry references. Essential for multi-step workflows like spot drill → peck drill on the same holes, or roughing → finishing with different tools and parameters.
- Right-click context menu: Duplicate, Delete, Enable/Disable, Move Up/Down
- Visual indicators: operation type icon, tool color coding, enabled/disabled state, time estimate

### 5.3 Properties Panel

When an operation is selected in the tree, the properties panel shows all editable parameters. Parameters inherited from the project or machine are shown with a subtle "inherited" indicator. Overriding a value makes it bold; a reset button restores inheritance.

### 5.4 Viewport

A 2D viewport built on QGraphicsView (or optionally OpenGL for performance).

#### 5.4.1 Navigation

- Pan (middle-mouse or Shift+drag), zoom (scroll wheel), fit-all (Home key)
- Stock outline shown as a dashed rectangle with origin marker
- Imported geometry rendered in layer colors
- Toolpath overlay: rapids in red dashed, feed moves in blue, arcs in green
- Z side-view inset: small panel showing the Z stack diagram
- Fixture/clamp overlay layer with collision highlighting

#### 5.4.2 Selection System

A rich selection system modeled after proven CAD/EDA tools (AutoCAD, KiCad, Altium):

- **Click select:** Click an entity to select it. Ctrl+click to toggle (add/remove from selection). Shift+click to add to selection.
- **Directional box select (AutoCAD/KiCad convention):** Drag left-to-right to select entities fully contained within the box (solid blue rectangle). Drag right-to-left to select entities touching or crossing the box (dashed green rectangle). This is an established convention that experienced CAD users will expect.
- **Select Similar:** Select one entity, right-click → "Select Similar." Options: same diameter (circles/arcs), same length (lines), same layer, same geometry type, same bounding-box size. This is critical for efficiency — selecting 200 identical mounting holes should be one action, not 200 clicks.
- **Select All on Layer:** Right-click → "Select All on Layer" to grab everything from a DXF layer.
- **Invert Selection:** Ctrl+I to invert the current selection.

#### 5.4.3 Measurement Tools

Built-in measurement overlay activated from the toolbar or keyboard shortcut. Measurements display directly on the viewport and do not require switching to another application.

- **Point-to-point distance:** Click two points to show the distance (and X/Y components). Snaps to entity endpoints, centers, midpoints, and intersections.
- **Entity dimensions:** Click a circle to show its diameter and center position. Click a line to show its length and angle.
- **Perimeter and area:** Select a closed contour to display its perimeter length and enclosed area.
- **Coordinate readout:** Continuous cursor position display in the status bar, in work coordinates.

### 5.5 Wizard System

Each wizard is a multi-step dialog that guides the user through creating an operation. Wizards generate operations in the tree; they do not replace the tree. After a wizard completes, every setting it configured is fully editable in the properties panel.

| Wizard | Steps | Output |
|---|---|---|
| Sheet Cutout | Select contours → Inside/Outside → Tool → Depth → Tabs → Lead-in/out | Profile operation with tabs |
| Pocket | Select regions → Tool → Depth → Strategy (zigzag/spiral) → Finishing pass? | Pocket operation |
| Drill Pattern | Select points or generate pattern → Bounding region → Tool → Depth → Cycle type | Drill operation |
| Surface Plane | Define area (stock or custom) → Tool → Stepover → Direction | Surface operation |
| Engrave / V-Carve | Select paths or enter text → Tool (V-bit) → Depth or auto-depth | Engrave operation |
| Multi-Step Drill | Select holes → Define steps (spot/pilot/final/tap) → Tool per step → Depths | Multiple drill ops, pre-ordered |
| Nesting | Select parts → Part count → Sheet size → Spacing → Rotation options | Nested layout with operations |
| Tile / Panelize | Select all ops → Tile grid → Overlap/registration | Tiling wrapper |

### 5.6 UX Philosophy: Zero-Ambiguity Interface

A core design goal is that the user should never need to consult external documentation to understand what a control does or what value to enter. Every dialog, wizard, and property field must be self-explanatory. This is achieved through four mechanisms:

#### 5.6.1 Self-Describing Controls

Every input field has a clear, unambiguous label that states exactly what the value means in practical terms. Labels avoid jargon where possible, and when technical terms are necessary (e.g., "stepover"), the label includes a brief clarification.

- **Bad:** "Stepover %" — stepover of what? Percentage of what?
- **Good:** "Stepover (% of tool diameter)" with the computed mm value shown live next to the field: "40% = 1.2 mm"
- **Bad:** "Depth" with a bare number field
- **Good:** "Target depth below surface" with the field showing "-6.0 mm" and a note: "(through stock: -6.0 mm)"

#### 5.6.2 Contextual Tooltips

Every control has a tooltip that answers three questions: What is this? What is a good value? What happens if I get it wrong? Tooltips should be concise (2–3 sentences maximum) and practical, not theoretical.

- **Bad tooltip:** "Sets the feed rate for the operation." — the user already knows that from the label.
- **Good tooltip:** "Horizontal cutting speed in mm/min. For a 3mm endmill in aluminum, 800–1500 is typical. Too fast risks tool breakage; too slow causes rubbing and heat buildup."

#### 5.6.3 Live Visual Feedback

Wherever possible, changing a value immediately updates the viewport preview. This makes abstract numbers concrete:

- Changing stepover shows the pocket fill pattern updating in real time
- Changing tab count shows tabs appearing/disappearing on the profile
- Changing lead-in style shows the entry path redrawing
- Changing depth updates the Z stack side-view diagram
- Out-of-range values highlight in yellow (warning) or red (error) with an inline explanation

#### 5.6.4 Sensible Defaults with Material Awareness

When a tool and material are selected, all numeric fields pre-populate with recommended values from the tool/material database. The user can accept them and proceed, or adjust. Defaults are clearly marked as "suggested" so the user knows they can change them. This eliminates the most common beginner question: "what number do I put here?"

#### 5.6.5 Integrated Contextual Help

Rather than a separate help manual, each wizard step and each properties panel section has an optional expandable "Learn more" area with a brief explanation and a diagram or illustration of the concept. For example, the lead-in/out step shows a small diagram of what arc, tangent, and direct entry paths look like. This is collapsed by default to keep the UI clean, but always one click away.

The goal is that a user with basic CNC knowledge (they understand what feeds, speeds, and depth-of-cut mean) can use every feature of the application without ever leaving it to read documentation.

### 5.7 Built-in Lightweight Simulator

The built-in simulator is not a full 3D material-removal engine. Instead, it provides three focused capabilities that catch the most common and dangerous mistakes before G-code ever reaches the machine.

- **Static Toolpath View:** All toolpaths overlaid on the geometry in the main viewport. Rapids shown as red dashed lines, feed moves in solid blue, arcs in green. This is always visible when toolpaths have been computed and provides an immediate visual sanity check.
- **Animated Playback:** A play/pause/step animation showing a tool marker moving along the entire toolpath in execution order. Speed slider for fast overview or slow inspection. A timeline scrubber allows jumping to any point. This reveals wrong operation order, unexpected rapids, missed areas, and direction issues (climb vs. conventional).
- **Simulation DRO (Digital Readout):** During playback, the status bar displays real-time position and machining state: X/Y/Z coordinates, feed rate, spindle RPM, current tool, and current operation name. A small tool profile graphic appears in the viewport corner showing the active tool. This keeps the display uncluttered while providing all the information a user needs to verify the simulation.
- **Pre-Flight Safety Dashboard:** A summary panel showing the Z stack diagram, travel envelope check, fixture collision overlay, and an overall go/warning/stop indicator. Warnings are clickable — clicking a warning highlights the offending operation and zooms to the problem area in the viewport.

For full 3D material-removal simulation, the external tools system (see below) provides integration with CAMotics or other simulators.

### 5.8 External Tools System

A generalized "launch pad" for integrating with external applications. Rather than building dedicated integrations, the system provides a configurable mechanism for launching any external tool with template-based argument substitution.

#### 5.8.1 Use Cases

- **3D Simulation:** Launch CAMotics with the generated G-code file for full material-removal visualization.
- **G-Code Sending:** Launch Universal G-Code Sender (UGS), bCNC, or CNCjs to stream G-code directly to the machine.
- **Geometry Editing:** Open the source DXF in LibreCAD or another editor for quick modifications, then re-import.
- **G-Code Inspection:** Open the generated G-code in a text editor for manual review.
- **Custom Scripts:** Launch user-written scripts for project-specific tasks (batch processing, custom validation, reporting).

#### 5.8.2 Configuration Format

```json
"external_tools": [
  {
    "name": "CAMotics Simulator",
    "path": "/usr/bin/camotics",
    "args": "{gcode_file}",
    "icon": "simulate",
    "show_in_toolbar": true,
    "category": "simulation",
    "platform_paths": {
      "linux": "/usr/bin/camotics",
      "windows": "C:/Program Files/CAMotics/camotics.exe",
      "macos": "/Applications/CAMotics.app/Contents/MacOS/camotics"
    }
  },
  {
    "name": "Universal G-Code Sender",
    "path": "/opt/ugs/bin/ugs",
    "args": "--open {gcode_file}",
    "show_in_toolbar": true,
    "category": "sender"
  }
]
```

#### 5.8.3 Template Variables

| Variable | Expands To | Example |
|---|---|---|
| `{gcode_file}` | Path to the generated G-code file | /home/user/project_output.nc |
| `{project_file}` | Path to the project file | /home/user/enclosure.pymcam |
| `{project_dir}` | Project directory | /home/user/projects/enclosure/ |
| `{dxf_file}` | Path to the imported DXF source | /home/user/enclosure.dxf |
| `{stock_x}`, `{stock_y}`, `{stock_z}` | Stock dimensions in current units | 300, 200, 6 |

## 6. Productivity Features

Beyond the core CAM operations, several productivity features significantly improve the day-to-day workflow. These are features that experienced users will expect and that beginners will appreciate once they discover them.

### 6.1 Feed & Speed Calculator

An integrated calculator that appears contextually during tool selection in wizards and in the properties panel. It is not a separate dialog — it is embedded in the workflow where feed/speed decisions are made.

- **Input:** Tool geometry (diameter, flute count, helix angle), material, and machine spindle range.
- **Output:** Recommended RPM, feed rate (mm/min), chip load per tooth, surface speed (m/min), and material removal rate.
- **Machine constraints:** Highlights when the optimal RPM exceeds the spindle range and automatically adjusts to the best achievable parameters. E.g., "Optimal: 24000 RPM. Your spindle max: 18000 RPM. Adjusted feed: 900 mm/min to maintain chip load."
- **Visual indicator:** A green/yellow/red "sweet spot" bar showing where the current parameters fall relative to the recommended range for the selected tool + material combination.

The calculator math is transparent — the formulas are shown so the user learns the relationships: chip load × flutes × RPM = feed, surface speed = π × diameter × RPM / 1000.

### 6.2 Operation Time Estimator

Machining time is computed automatically for every operation from the toolpath length and feed rates. Times are displayed directly in the operations tree (see section 5.2). The estimator includes:

- Cutting time: total feed-move distance divided by feed rate
- Rapid time: total rapid-move distance divided by machine rapid rate (from machine definition)
- Tool change time: configurable per machine (default 90 seconds for manual change, includes probing if configured)
- Dwell time: sum of all programmed dwells (peck drill retract, etc.)
- Total project time: sum of all operations plus tool changes, shown at the tree root

This makes the time cost of every decision visible. Adding a finishing pass? You can see it adds 12 minutes. Reordering to eliminate a tool change? The total drops by 90 seconds immediately.

### 6.3 DXF Layer-to-Operation Mapping

A convention-based quick-setup feature for users who design with CNC in mind. During DXF import, layer names are parsed for operation hints and the system offers to auto-create operations.

- **Convention:** Layer names follow a pattern like "Profile_Outside", "Pocket_3mm", "Drill_2mm", "Engrave". The import wizard recognizes the prefix (operation type) and optional suffix (tool size or parameter).
- **Auto-mapping dialog:** After import, a dialog shows detected mappings and lets the user confirm, adjust, or ignore each one. Unrecognized layers are imported as plain geometry.
- **Custom mapping rules:** Users can define their own naming conventions in preferences (e.g., a company-standard layer naming scheme).

This creates an extremely fast workflow: design in CAD with layer discipline, import into PyMillCAM, review the auto-generated operations, adjust parameters, generate G-code. For repeat jobs with established conventions, this can reduce CAM setup to seconds.

### 6.4 Nesting / Layout Optimization

When cutting multiple parts from a single sheet, automatic nesting arranges them to minimize material waste. The nesting system operates on 2D profile contours.

- **Input:** Part profiles (from geometry or existing profile operations), part count, sheet dimensions, minimum spacing between parts, and allowed rotations (0°, 90°, 180°, 270°, or free).
- **Rectangular nesting (v1):** Bounding-box-based bin packing. Fast, simple, covers most hobby use cases. Reports material utilization percentage.
- **True-shape nesting (v2):** NFP (No-Fit Polygon) based nesting that follows actual contour shapes. Dramatically better for complex or irregular parts. Higher computational cost.
- **Output:** Positioned part instances with their operations. Each instance is a reference to the original operations, transformed to the nested position. Tab and lead-in positions are recalculated per instance.

### 6.5 Toolpath Optimization

After all operations are computed, a global optimization pass reduces total machining time without affecting the cut result:

- **Tool grouping:** Operations are automatically grouped by tool to minimize tool changes. Within each tool group, the original operation order is preserved unless the user enables reordering.
- **Rapid minimization:** Within each tool group, the order of individual operations (and within operations, the order of disconnected regions) is optimized to minimize total rapid-travel distance using a nearest-neighbor heuristic or simple TSP solver.
- **Drill path optimization:** For drill operations with many holes, the drilling order is optimized to minimize total travel. This can be significant — an optimized path through 200 holes can be 50–70% shorter than an unoptimized one.
- **Preview:** Before/after visualization showing the original and optimized rapid paths with time savings displayed.

### 6.6 Undo / Redo System

A full command-pattern undo/redo system that covers every user action. This is not just text-field undo — it tracks all project-level changes:

- Operation add / remove / duplicate / reorder
- Parameter changes (depth, feed, RPM, stepover, etc.)
- Geometry assignment to operations (add/remove paths)
- Tool and tool controller changes
- Fixture/clamp add / remove / move
- Project settings changes (stock size, origin, Z reference)

The undo stack is unbounded within a session and is cleared when the project is closed. Every action is a discrete command object that can be undone and redone. The current undo state is shown in the Edit menu and accessible via Ctrl+Z / Ctrl+Shift+Z.

### 6.7 Quick Duplicate and Array

Two related features for creating copies of operations at different positions:

- **Quick Duplicate:** Select one or more operations, Ctrl+D to duplicate. All geometry references are preserved. The user can then modify the duplicate (change tool, change depth, change cycle type) without affecting the original. Primary use case: multi-step machining on the same geometry (spot drill → peck drill → ream on the same holes).
- **Linear / Grid Array:** Select operations, invoke Array, specify count and spacing (or grid rows × columns). Creates copies of the geometry and operations at the specified positions. Useful for production runs of identical features (e.g., "4 of this pocket spaced 50mm apart").

### 6.8 G-Code Preview with Bidirectional Linking

Before final export, the generated G-code is shown in a syntax-highlighted text panel. This is not a generic text editor — it has bidirectional linking with the viewport:

- Clicking a line in the G-code highlights the corresponding move in the viewport and scrolls to it
- Clicking a toolpath segment in the viewport scrolls the G-code panel to the corresponding line and highlights it
- Operations are visually separated in the G-code with collapsible sections
- Syntax highlighting distinguishes rapids (G0), feed moves (G1), arcs (G2/G3), tool changes, spindle commands, and comments
- Line count and estimated time are shown per section

This is a read-only preview — the G-code is generated, not hand-written. If the user needs to edit, the external tools system can open it in a text editor.

### 6.9 Startup Page and Project Templates

When the application launches (or when no project is open), a startup page is shown with quick-start options:

- **Recent Projects:** List of recently opened projects with thumbnail previews, sorted by last modified date.
- **New Blank Project:** Opens the machine/stock setup dialog to start from scratch.
- **New from Template:** Pre-configured project templates for common tasks: "Sheet cutout with tabs", "Enclosure top (pockets + drilling)", "Sign/engraving", "Drill jig." Templates include example stock sizes, operation types, and suggested tool configurations.
- **Import DXF:** Direct shortcut to import a DXF and start a new project from it.

## 8. Development Roadmap

The roadmap is structured in phases, each producing a usable (if limited) tool. This ensures value at every stage rather than a long wait for a complete product. Time estimates assume development with an AI coding assistant (Claude Code), which significantly accelerates scaffolding, boilerplate, data models, and test generation. The algorithmic core (toolpath geometry, offset computation) benefits less from AI assistance and requires more iterative testing.

| Phase | Solo Developer | With Claude Code | Compression |
|---|---|---|---|
| 1: Foundation | 4 weeks | 1.5–2 weeks | ~50% |
| 2: Core Operations | 4 weeks | 2–2.5 weeks | ~40% |
| 3: Wizards & Safety | 4 weeks | 2–3 weeks | ~35% |
| 4: Advanced Features | 4 weeks | 2.5–3 weeks | ~30% |
| 5: Polish & Ecosystem | 5 weeks | 2.5–3 weeks | ~45% |
| **Total** | **21 weeks** | **10.5–14 weeks** | **~40%** |

The biggest gains come from phases heavy on data models, UI scaffolding, and boilerplate (Phases 1, 2, 5). The smallest gains are in Phase 4 where the geometry algorithms require careful mathematical reasoning and visual debugging.

### Phase 1: Foundation (1.5–2 weeks)

**Goal:** A minimal tool that can import a DXF, assign a profile operation to a contour, and generate G-code.

- Core data models with Pydantic (Project, Stock, Operation, Tool, Machine)
- DXF import via ezdxf (lines, arcs, polylines, circles)
- Basic profile toolpath generation (outside/inside offset, multi-depth)
- IR definition and one post-processor (UCCNC)
- Minimal PySide6 window: viewport + simple tree + output panel
- Directional box selection (left-to-right contained, right-to-left crossing)
- Undo/redo command infrastructure (command pattern, stack, Ctrl+Z/Ctrl+Shift+Z)
- Coordinate readout in status bar
- Project save/load as JSON

### Phase 2: Core Operations (2–2.5 weeks)

**Goal:** A working tool for the most common sheet operations with smart selection and time awareness.

- Pocket toolpath (zigzag strategy)
- Drill operation (simple and peck cycles)
- Tab generation for profile operations
- Lead-in / lead-out for profiles
- Tool library (create, edit, save, load)
- Machine definition system with defaults cascade
- Select Similar (same diameter, same layer, same geometry type)
- Operation time estimator with tool change time in tree
- Integrated feed/speed calculator (contextual, in tool selection flow)
- Operation duplication (Ctrl+D with geometry preservation)
- Measurement tools (point-to-point distance, entity dimensions)
- Second post-processor (Mach3)

### Phase 3: Wizards & Safety (2–3 weeks)

**Goal:** Wizard-driven workflow, pre-flight safety, and built-in simulation.

- Profile Cutout Wizard with step-by-step dialog
- Pocket Wizard
- Drill Pattern Wizard (with bounding region)
- Multi-Step Drilling Wizard (spot → pilot → final → tap)
- DXF layer-to-operation mapping (convention-based auto-setup)
- Z stack model and budget calculation
- Pre-flight safety checks (reachability, safe height, XY travel)
- Built-in lightweight simulator (static view + animated playback + safety dashboard)
- Properties panel with cascade visualization (inherited vs. overridden)
- Fixture/clamp layer with collision detection

### Phase 4: Advanced Features (2.5–3 weeks)

**Goal:** Surface operations, engraving, and pattern generation.

- Surface / facing operation
- Engrave operation (path following, fixed depth)
- V-carve (depth from width + V-angle calculation)
- Pattern generators (rectangular grid, hex grid, circular array)
- Pattern clipping to bounding geometry (CamBam-style vent grids)
- Text-to-geometry conversion for engraving
- Additional post-processors (GRBL, LinuxCNC)

### Phase 5: Polish & Ecosystem (2.5–3 weeks)

**Goal:** Production-quality UX, interoperability, optimization, and nesting.

- Nesting / layout optimization (rectangular bin-packing v1)
- Toolpath optimization (tool grouping + rapid minimization + drill path TSP)
- Quick duplicate and linear/grid array
- External tools system (configurable launcher with template variables)
- CAMotics integration as default external simulator
- Auto-detection of installed external tools (CAMotics, UGS, bCNC)
- FreeCAD .fctb/.fctl tool library import
- LinuxCNC tool table import
- Material database with wizard integration
- Tiling / panelization for oversized parts
- 2.5D contour operations
- Keyboard shortcuts and workflow optimization
- Packaging (PyInstaller or Nuitka for Windows/Mac/Linux)

## 9. Open Questions & Future Considerations

Items that need further discussion or research before implementation:

- **Geometry kernel choice:** Shapely + pyclipper is the pragmatic starting point, but if 3D support is ever desired, Open CASCADE (via CadQuery/build123d) would be the path. Should the geometry abstraction layer anticipate this?
- **Plugin / scripting system:** CamBam's extensibility via scripts was a major strength. Should PyMillCAM support user plugins (Python scripts that register as new operations, wizards, or post-processors)?
- **Simulation depth:** The built-in 2D simulator and external CAMotics integration cover most needs. If demand arises, a lightweight 2.5D height-map renderer (showing material removal from the top view as a color gradient) could be an intermediate step before full 3D — simpler than CAMotics but more informative than 2D lines.
- **Multi-setup / flip operations:** Machining both sides of a part requires coordinating two setups with registration features. Is this in scope?
- **Image-to-toolpath:** Estlcam can trace bitmaps. This is useful for engraving images. Priority TBD.
- **Cloud / sharing:** Should tool libraries, machine definitions, or material databases be shareable via an online repository?
- **Licensing:** Open-source license selection (GPL vs. LGPL vs. MIT). Implications for using Shapely (BSD), pyclipper (Boost), PySide6 (LGPL), ezdxf (MIT).
- **Trochoidal / adaptive clearing:** Circular sweeping toolpaths that reduce tool load significantly, especially in metals. Fusion 360's adaptive clearing is one of its most praised features. Algorithmically complex but very valuable for aluminum and harder materials. Candidate for a later phase.
- **True-shape nesting:** NFP (No-Fit Polygon) based nesting that follows actual contour shapes rather than bounding boxes. Significant algorithm complexity but dramatically better material utilization for irregular parts.
